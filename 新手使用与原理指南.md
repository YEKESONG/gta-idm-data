## 1. 这个工具是干什么的

一句话：**自动从 YouTube / B 站抓 GTA V（侠盗猎车手 5）的游戏实况视频，自动剪掉没用的部分，
留下一堆"人物在持续移动、开火"的干净短片**，用来给 AI 模型当训练素材。

### 为什么要这些数据？

研究者想训练一种叫 **IDM（逆动力学模型，Inverse Dynamics Model）** 或 **world model（世界模型）**
的 AI。简单说，就是让 AI"看视频学会游戏里的动作和物理规律"。要学得好，就需要**大量**
画面里**真的在发生有意义的动作**（跑动、射击）的视频。

但网上的实况视频里掺了太多"废料"：

- 角色死亡后的 **WASTED / BUSTED** 大字结算画面；
- 打开**菜单、地图、武器商店**的界面；
- **读条、黑屏、转场**；
- 角色**站着不动**发呆的片段。

这些对训练没价值，甚至有害。人工一帧帧挑出来要命，所以这个工具把整个"找→下→切→挑"
的过程**全自动化**了：你给几个关键词或者几个视频链接，它就能滚动产出**几百小时**清洗好的素材。

### 这个工具适合谁？

- 想为 AI 训练**批量准备视频数据集**的人；
- 想学习**爬虫 + 视频处理流水线**怎么设计的人。

> ⚠️ **合规提示**：本工具仅用于学术研究的数据准备。抓取前请确认目标平台的服务条款与
> 版权要求，控制抓取频率，别给网站造成压力。

## 2. 整体工作流程（一张图看懂）

整个工具就是一条**流水线（pipeline）**，像工厂流水线一样，原料从一头进去、成品从另一头出来：

```
  关键词/链接                                                          干净短片
      │                                                                  ▲
      ▼                                                                  │
  ① 发现        ② 下载         ③ 切片          ④ 过滤           ⑤ 裁剪保存
 discover  ──► download  ──►  segment  ──►   filter    ──►    export
 找候选视频    用 yt-dlp 下    把长视频切成   逐帧检查，判断    通过过滤的片段才
（只拿信息，    成本地文件     一段段小片段   每段是好是坏      真正用 ffmpeg 剪出来
 不下视频）   （最高 480p）  （只标时间点）  （4 个检测器）    （并写进清单 manifest）
```

5 个阶段每个都是 `gta_pipeline/` 文件夹下的一个独立 Python 文件，既能单独调试，
也由总指挥文件 `gta_pipeline/pipeline.py` 串起来一口气跑完。

**一个关键的省力设计**：第 ③ 步切片时**并不真的去剪视频文件**，只记下"从第几秒到第几秒"
这样的时间区间；只有第 ④ 步过滤**通过**的片段，第 ⑤ 步才真的动手裁剪。
这样就不会为了最后要扔掉的片段白白浪费时间和磁盘。

## 3. 安装：从零把环境装好

这个工具用 **Python** 写，依赖几个外部程序。下面假设你用的是 **Linux（如 Ubuntu）**。

### 第 1 步：装 conda（如果还没有）

conda 是管理 Python 环境的工具，能让这个项目的依赖和你电脑上其它项目互不打架。
如果你已经装过 Anaconda 或 Miniconda 可跳过。没装的话去搜 "Miniconda 安装"，按官网装即可。

### 第 2 步：创建并激活项目专用环境

> 📌 **重要**：本项目必须用名为 `gta_idm_data` 的 conda 环境。它里面装了 `yt-dlp`、
> `opencv`、`scenedetect` 这些关键库。如果用别的环境（比如默认的 `ml`），会因为缺
> `yt_dlp` 而跑不起来。

```bash
# 激活项目环境（假设已经建好；没建过就先 conda create -n gta_idm_data python=3.10）
conda activate gta_idm_data

# 装 Python 依赖
pip install -r requirements.txt
```

### 第 3 步：装 ffmpeg（系统级，必装）

**ffmpeg** 是处理音视频的"瑞士军刀"。下载视频时要靠它合并/转码，裁剪片段时也要靠它。

```bash
sudo apt install ffmpeg          # Debian / Ubuntu
```

装完后验证一下（应该能打印版本号）：

```bash
ffmpeg -version
yt-dlp --version
```

> 没有 ffmpeg 也能勉强下载，但无法统一帧率、无法物理裁剪片段，强烈建议装上。

## 4. 上手：3 条命令跑出第一批数据

> 全程在项目根目录 `gta-idm-data/` 下、并且已经 `conda activate gta_idm_data` 的前提下操作。

### 命令 1：先"试跑一个视频"，确认链路通

第一次用，建议先用最小的配置只处理 1 个视频，跑通整条流水线：

```bash
python -m scripts.run_pipeline --config configs/test_single.yaml
```

如果它能下载、切片、过滤、最后在 `data/clean/` 里产出几个 `.mp4`，说明环境没问题。

### 命令 2："干跑"看看会抓到哪些视频（不下载）

正式大批量跑之前，强烈建议先**只看候选列表**，确认关键词和排序对不对。
`--discover-only` 表示"只发现、不下载、不过滤"，几十秒就能跑完：

```bash
python -m scripts.run_pipeline --config configs/youtube_300h.yaml --discover-only
```

它会打印类似这样的列表（平台、时长、标题、链接）：

```
发现 312 个候选（bilibili 278，youtube 34）：
  [bilibili]      ?  GTA5 主线剧情流程 第01集 ...  https://www.bilibili.com/...
  [youtube]   1820s  GTA V Free Roam No Commentary ...  https://www.youtube.com/...
  ...
```

> B 站视频显示时长为 `?` 是正常的（它在"只拿信息"模式下不返回时长），不影响下载——
> 下载前工具会自动再探测一次时长。

### 命令 3：正式跑

确认候选没问题，就正式开跑。这一步会真的下载、切片、过滤、保存：

```bash
python -m scripts.run_pipeline --config configs/youtube_300h.yaml
```

- 这会**跑很久**（目标是凑满 250 小时有效素材），但你**随时可以按 `Ctrl+C` 中断**。
- 下次再运行**同一条命令**会**自动断点续跑**：已经处理过的视频会跳过，已累计的时长会接着算，
  不会重头再来。

### 命令行开关速查

| 开关 | 作用 |
|------|------|
| `--config <yaml>` | 指定配置文件。不给则用一套内置默认值 |
| `--discover-only` | 只跑发现阶段、打印候选（含平台分布和排序），不下载/不过滤 |
| `--refresh-discovery` | 忽略候选缓存，强制重新搜索（一般不需要，见 [6.1](#61-怎么发现候选视频discovery)） |

## 5. 产物：跑完之后我得到了什么

所有产物都在配置里 `work_dir` 指定的目录下（默认 `data/`）：

| 路径 | 内容 |
|------|------|
| `data/clean/*.mp4` | ⭐ **最终成品**：通过过滤的干净短片（固定帧率、无音轨） |
| `data/manifest.jsonl` | ⭐ **元数据清单**：每个保留片段一行 JSON，记录来源、时间区间、帧率、过滤统计 |
| `data/processed.txt` | 已处理完的视频清单（断点续跑用，每行 `平台_id ⇥ 源视频时长`） |
| `data/discovery_cache.jsonl` | 候选列表缓存（重跑直接复用，不再发网络请求） |
| `data/discovery_cache.jsonl.seeds` | 已展开过的"种子 URL"记录（增量采集用） |
| `data/raw/` | 下载的原始视频（配置 `keep_raw: false` 时，处理完即删，省磁盘） |

### `manifest.jsonl` 长什么样？

每行是一个 JSON，例如：

```json
{"clip_id": "youtube_e8WaLAMoyNQ_clip0000", "source_video": "youtube_e8WaLAMoyNQ.mp4",
 "platform": "youtube", "start_s": 0.0, "end_s": 18.967, "fps": 20,
 "exported_path": "data/clean/youtube_e8WaLAMoyNQ_clip0000.mp4",
 "decision_reason": "ok", "bad_frame_ratio": 0.0, "static_frame_ratio": 0.026,
 "label_counts": {}}
```

字段含义：

- `clip_id`：这个片段的唯一编号；
- `source_video` / `platform`：它来自哪个视频、哪个平台；
- `start_s` / `end_s`：它在源视频里的起止秒数；
- `fps`：帧率（统一成 20，方便按帧对齐动作标签）；
- `exported_path`：剪出来的 mp4 存在哪；
- `decision_reason`：为什么保留/丢弃（`ok` 表示通过）；
- `bad_frame_ratio` / `static_frame_ratio`：坏帧、静止帧的占比（过滤的依据）；
- `label_counts`：各类坏画面各出现了多少帧（便于分析数据集质量）。

训练 AI 时，PyTorch 的 `Dataset` 可以**直接一行行读这个清单**，按时间区间取帧，非常方便。

## 6. 原理详解（核心）

这一章逐个阶段讲清楚"它到底怎么做到的"。即使你不懂代码也能看懂思路。

### 总览：筛选分两层

工具要从茫茫视频里挑出好素材，用了**两层筛选**，从便宜到昂贵：

1. **发现层（粗筛）**：只看**便宜的信息**——标题、平台、时长。这一步不下视频，
   只看"标题像不像 GTA、是不是太长"，先把明显不要的挡掉。
2. **过滤层（精筛）**：把视频真正下载下来，**一帧帧看画面内容**，判断每个片段是
   "正常游戏"还是"死亡/菜单/读条/发呆"。这一步贵，但准。

下面按阶段展开。

---

### 6.1 怎么"发现"候选视频（discovery）

📄 对应代码：[`gta_pipeline/discovery.py`](gta_pipeline/discovery.py)

**目标**：把"几个关键词 / 几个链接"变成一批**候选视频引用（VideoRef）**——只是一份
"视频清单"（标题、链接、时长），**不下载视频本身**。这样可以先按时长等便宜信号粗筛，
再决定下载哪些，省带宽。

**两种来源**：

1. **关键词搜索**：靠 `yt-dlp` 这个万能下载器自带的搜索功能。
   - YouTube 用 `ytsearch` 前缀，B 站用 `bilisearch` 前缀；
   - 用的是 "flat（扁平）模式"：只解析出一个列表、拿到每条的元数据，**不深挖每个视频**，所以很快。
2. **种子 URL（seed_urls）**：你**手动精选**的频道、播放列表、合集链接。
   `yt-dlp` 会自动把一个合集链接**展开成里面的全部视频**。
   （像 Twitch 这种没有公开搜索接口的平台，就只能靠这种方式。）

**几个关键设计**：

- **标题白名单（`require_title_keywords`）**：只对"关键词搜索"的结果做标题过滤——
  比如要求标题里必须含 `GTA V` 或 `GTA 5`，给搜索质量兜底。而 `seed_urls` 是你
  人工精选的可信来源，**不做标题过滤，全部保留**。
- **平台优先级（`platform_priority`）**：让某个平台的候选排到队首先处理。
  例如 `[bilibili, youtube]` 表示先把 B 站精选合集全采集完，再去 YouTube。
- **缓存 + 增量种子**：搜出来的候选会写进 `discovery_cache.jsonl`。
  - 下次重跑会**直接复用缓存，一个网络请求都不发**（这是断点续跑时最有效的"防限流"手段）。
  - 但如果你往配置里**新加了 seed_urls**，工具会**只增量展开这些新链接**并入缓存
    （已展开过的记在 `.seeds` 小文件里，不重复展开，也不重新搜整轮关键词）。
    所以你随时可以往配置贴新合集链接，直接重跑就会采集到。
- **B 站多 P 视频的坑**：B 站一个"BV 号"下可能有几十个分 P（分集），它们共用同一个
  视频 id，只靠链接里的 `?p=N` 区分。代码特意从链接里解析出"BV 号 + 分 P 号"
  当作唯一标识，否则几十集会被去重当成一集，最后只下到第一集。

> 💡 **防限流**：发现阶段每个网络请求之间会**随机等待 2~4 秒**；万一撞上 429
> （请求太频繁被限流），会**指数退避重试**（等 5 秒、10 秒、20 秒……）而不是
> 默默丢掉，避免"假装跑完但其实啥也没抓到"。

### 6.2 怎么下载视频（download）

📄 对应代码：[`gta_pipeline/download.py`](gta_pipeline/download.py)

**目标**：把候选视频下载成本地 mp4，并统一规格。

**关键点**：

- **画质上限 480p**：`max_height: 480`。对 world model 训练来说 480p 足够
  （后面往往还会再缩到 256），却能大幅省带宽和磁盘。
- **优先 H.264（avc1）编码**：YouTube 现在很多视频是 AV1 编码，而 OpenCV 自带的解码器
  解不了 AV1（会刷 "Failed to get pixel format" 报错）。所以下载时优先选 avc1 编码，
  YouTube 在 ≤720p 几乎总有 avc1 可选。
- **下载前先按时长粗筛**：`max_duration_s`（默认配置里设 3 小时）。十几、二十小时的
  超长直播录像会被切出**几千个片段**，极易把整条流水线拖死（曾经卡过 2.5 小时没进展），
  而且单个文件可达 10GB+。所以这种超长视频**在下载前就按时长跳过**，靠数量去凑总时长。
  - **例外**：如果某个超长视频**之前已经完整下到本地了**，则豁免这个上限、优先处理掉，
    不浪费已经花掉的下载流量。
- **固定帧率**：注意帧率统一**不在下载这步做**，而是留到第 ⑤ 步裁剪时用 ffmpeg 的 `-r`
  统一成 20fps。固定帧率很重要：训练时要"按第几帧"给动作打标签，帧率不一致会让
  "第 t 帧"对应的真实时间错位。
- **礼貌抓取**：`sleep_interval_s` 让每个视频下载之间随机等待几秒，降低被站点限流的风险。
- **容错**：单个视频下载失败只是跳过它（并清掉残留的半成品文件），不会中断整批。

### 6.3 怎么把长视频切成片段（segment）

📄 对应代码：[`gta_pipeline/segment.py`](gta_pipeline/segment.py)

**目标**：把一段几十分钟、几小时的长视频，切成一段段几秒到几十秒的小片段（Clip）。

**两种切法**：

1. **`scene`（按镜头切，默认且推荐）**：用 **PySceneDetect** 库的 `ContentDetector`
   检测"画面内容突变"的地方作为镜头边界。
   - 为什么这样切聪明？因为 GTA 实况里"进商店 / 死亡 / 读条"通常都伴随**明显的画面切换**，
     按镜头切能让这些坏段落**自然地落在独立的片段里**，方便后面整片一起丢掉。
   - `scene_threshold`（默认 27）是灵敏度：越小越容易判定为"换镜头"、切得越碎。
2. **`window`（固定时长窗口）**：简单粗暴，每隔固定秒数切一刀。适合连续的自由探索录像。

**时长约束**：

- 太短（< `min_clip_seconds`，默认 2 秒）的碎片直接丢；
- 太长（> `max_clip_seconds`，默认 30 秒）的片段再均匀切成小段。

> 🔑 **最重要的设计**：这一步**根本不碰视频文件**，只产出一组 `(起始秒, 结束秒)` 的
> **时间区间**。真正的剪辑留到"过滤通过之后"再做（见 [6.5](#65-怎么裁剪并保存export)），
> 避免给最后要丢掉的片段白白写盘。

**抽帧采样**：过滤的时候**不需要逐帧检查**。函数 `sample_frames` 按 `sample_fps`
（默认每秒 2 帧）在片段时间区间里采样几帧就够了——既快，又足以判断这个片段属于哪类内容。
它用 OpenCV 的时间定位（`CAP_PROP_POS_MSEC`）直接跳到某个时间点取帧，不用把整段读进内存。

### 6.4 怎么筛选出"好"片段（4 个检测器）（filter）

📄 对应代码：[`gta_pipeline/filters/`](gta_pipeline/filters/) 和 [`gta_pipeline/policy.py`](gta_pipeline/policy.py)

这是整个工具**最核心**的部分。它分成两步、职责清晰：

> **第一步：检测器逐帧打分 → 第二步：策略汇总成整片决策**

#### 第一步：4 个检测器，每个只管一类坏画面

每个采样帧都会过一遍所有检测器，每个检测器给出一个判断（标签 + 分数）。
它们都用的是**轻量的图像启发式规则**（不需要 GPU、不需要训练模型），又快又好解释：

| 检测器 | 想丢掉什么 | 怎么判断（大白话） |
|--------|-----------|-------------------|
| **`death_screen`** | WASTED / BUSTED / 任务结算 | GTA 死亡画面有两个特征：① 整屏**去饱和**（接近灰色，引擎特效）；② 屏幕中央有一行**很大的高对比文字**。两者同时出现就判为 `death`。装了 OCR（pytesseract）的话还会读中央文字确认关键词，更准。 |
| **`menu_shop`** | 菜单 / 商店 / 暂停 / 地图 | 菜单界面有大片**平坦的纯色区域**（面板、遮罩），自然纹理少；而正常游戏画面（街道、建筑、人物）纹理丰富。用 Sobel 梯度算"平坦像素占比"，占比太高就判为 `menu`。 |
| **`loading_black`** | 黑屏 / 读条 / 转场 | 读条和转场画面要么**整屏极暗**（平均亮度很低 → 黑屏），要么**整屏几乎一种颜色**（亮度方差极低 → 纯色过渡）。命中就判为 `loading`。 |
| **`motion`** | 长时间站着不动 | 我们要的是"移动 + 开火"。用 **Farneback 稠密光流**算相邻两帧每个像素移动了多少，取平均幅度作为"运动强度"。低于 `min_motion_magnitude`（默认 0.6）就判为 `static`（静止）。 |

> 💡 这几个检测器都是"**有状态可重置**"的设计：`motion` 需要记住上一帧才能算光流，
> 所以每处理完一个片段，工具会调用 `reset()` 把"上一帧"清掉，避免串片。

> 💡 检测器是**可插拔**的：在 `filters/` 下新建一个文件、继承 `FrameDetector`、用
> `@register` 注册，再在配置 `enabled_detectors` 里写上名字就能启用。还预留了
> **模板匹配**（把死亡/菜单截图放进 `assets/templates/`）和 **ML 分类器**
> （`filters/classifier.py`，用 ResNet18 兜底难分画面）两个增强钩子。

#### 第二步：策略层汇总成"保留还是丢弃"

📄 [`gta_pipeline/policy.py`](gta_pipeline/policy.py) 的 `decide_clip` 函数。

为什么要单独一层？因为"单帧是好是坏"和"多少坏帧才丢整片"是两个不同的问题。
分开后，既能复用同一批检测器去试不同的阈值，也能讲清楚每个片段**为什么**被丢。

规则（针对一个片段的所有采样帧统计）：

- **坏帧**：只要任一检测器把这帧判为 `death` / `menu` / `loading` / `non_gameplay`，这帧就算坏帧；
- **静止帧**：`motion` 把这帧判为 `static`；
- 然后：
  - **坏帧占比 > 25%**（`bad_frame_ratio_to_drop`）→ 丢掉整片（死亡/菜单/读条太多）；
  - **静止帧占比 > 60%**（`static_frame_ratio_to_drop`）→ 丢掉整片（缺少移动/开火动作）；
  - 都没超 → **保留**。

注意 `static` 本身不算"坏帧"——一个片段里偶尔几帧静止没关系，只有**静止帧太多**才丢。
这个判断由策略层按比例决定，检测器自己不下结论。

### 6.5 怎么裁剪并保存（export）

📄 对应代码：[`gta_pipeline/pipeline.py`](gta_pipeline/pipeline.py) 的 `export_clip` 函数。

只有**过滤通过**的片段，才会走到这一步真正被剪出来：

- 用 **ffmpeg** 按之前记下的时间区间裁出独立 mp4；
- `-ss`（起始时间）放在 `-i`（输入）**前面**，做"快速跳转"，对超长源文件也能秒级定位；
- `-r 20` **固定帧率**，保证最终片段都是 20fps（按帧对齐动作标签）；
- `-an` **去掉音轨**（动作建模用不到声音，省空间）；
- 裁剪带 **300 秒超时**：正常裁一个 ≤30 秒的片段只要约 1 秒，但个别超长源文件可能让
  ffmpeg 卡死，超时就杀掉它、跳过这个片段，不会阻塞整条流水线。

剪好之后，这个片段的元数据就被写进 `manifest.jsonl`（清单只在主线程串行写，避免多进程抢写冲突）。

> 如果系统里没装 ffmpeg，工具不会硬剪，而是退化成"清单里只记源视频 + 时间区间"，
> 训练时按区间去源视频里读帧也能用。

### 6.6 怎么做到又快又稳（并行流水线）

📄 对应代码：[`gta_pipeline/pipeline.py`](gta_pipeline/pipeline.py) 的 `run` 函数。

如果一个一个视频"下完→切完→过滤完"串着干，会有大量空等：切片时下载闲着，过滤时切片闲着。
所以工具设计成**三级流水线**，让三种不同的资源**同时干活、互不空等**：

```
 下载线程 ×N (网络IO)  ──dl_q──►  切片线程 ×1 (单核场景检测)  ──seg_q──►  过滤进程池 ×M (多核)
 download_concurrency              PySceneDetect                          num_workers
```

- **重叠执行**：当"视频 N"在过滤时，"视频 N+1"已经在下载和切片了，消除了空转浪费。
- **有界队列做"背压"**：两级之间用容量受限的队列（`dl_q` / `seg_q`）连接。
  下载比处理快时，队列一满下载线程就自动阻塞，把磁盘占用限制在少数几个"在途"视频，
  适合大批量抓取不爆盘。
- **避免抢 CPU（超额订阅）**：过滤是**多进程**并行（每个进程吃一个核），所以要把 OpenCV
  自己的内部多线程关掉（`cv2.setNumThreads(1)`），否则"进程数 × 每进程线程数"会互相抢核反而更慢。
- **⚠️ 进程池必须用 `spawn` 而不是 `fork`（关键踩坑）**：主进程这时已经起了下载/切片线程，
  此时若用 `fork` 复制出工作进程，工作进程会**继承"别的线程正持有、却永远不会释放"的底层锁**
  （malloc / OpenCV / ffmpeg 的全局锁），结果工作进程一去拿锁就**永久死锁**，整条流水线挂起
  （曾卡死 2.5 小时）。`spawn` 启动一个全新的 Python 解释器、不继承父进程的线程和锁状态，
  从根上消除了这个死锁。
- **串行写**：`manifest.jsonl` 和 `processed.txt` 只由主线程写，避免多进程并发写冲突。

> 这些并行参数（`download_concurrency` / `download_prefetch` / `segment_prefetch` /
> `num_workers`）都能在配置里调。磁盘紧张就把预取数调成 1；想更快就把进程数调大。

### 6.7 怎么断点续跑、什么时候自动停

- **断点续跑**：每处理完一个视频，就把它追加进 `processed.txt`。重跑时按这个清单
  **跳过已完成的视频**，不会重复选中、重复下载。已经下到本地的文件，yt-dlp 也会
  自动跳过下载直接复用。
- **按"有效时长"自动收口**：配置里的 `target_hours` 统计的是**过滤后保留下来的干净片段
  总时长**，而**不是**源视频时长——因为源视频经过切片+过滤后产出率只有约 40%，按源时长
  算会严重不足。累计达到目标（如 250 小时）就自动停止，不必跑完所有候选。续跑时基线
  从已有的 `manifest.jsonl` 累加，跨多次重跑也能正确累计。
- **容错**：单个视频在下载/切片/过滤任一步报错，都只跳过它，不中断整批。

## 7. 配置文件怎么改

配置文件在 [`configs/`](configs/) 下，是 YAML 格式（一种很好读的文本格式）。
现成的有：

- `configs/test_single.yaml` — 只处理 1 个视频，第一遍跑通用；
- `configs/youtube_300h.yaml` — ⭐ **主力配置**（B 站合集优先 + YouTube 搜索，凑 250h）；
- `configs/batch.yaml` — 关键词驱动的通用批量模板；
- `configs/default.yaml` — 全套默认值。

以主力配置为例，逐项注释最常改的参数：

```yaml
work_dir: data                # 所有产物放在这个目录
target_hours: 250             # 凑满 250 小时"有效（干净片段）时长"就停（不是源视频时长！）
keep_raw: false               # 处理完即删原始视频，省磁盘
platform_priority: [bilibili, youtube]   # 先采 B 站合集，再采 YouTube
num_workers: 0                # 过滤进程数，0 = 自动（按 CPU 核数留余量，上限 12）

discovery:
  platforms: [youtube]        # 只对 YouTube 做关键词搜索（B 站搜索质量差，只用 seed_urls）
  queries:                    # 搜索关键词；GTA 纯画面建议带 "no commentary"（无解说）
    - "GTA V gameplay no commentary"
    - "GTA 5 free roam shooting"
  max_results_per_query: 100  # 每个关键词最多取多少个候选
  require_title_keywords: ["GTA V", "GTA 5"]   # 标题白名单，只对"搜索结果"生效
  seed_urls:                  # 精选链接，不受标题过滤，全部保留；B 站多 P 合集会展开成全部分 P
    - "https://www.bilibili.com/video/BV1indnBREvD"
    - "https://www.youtube.com/playlist?list=..."

download:
  max_height: 480             # 画质上限 480p
  fps: 20                     # 最终干净片段统一帧率
  max_duration_s: 10800       # 单视频时长上限 3 小时，超过的下载前就跳过
  sleep_interval_s: 5.0       # 每个视频下载间随机等待，降低被限流风险

segment:
  mode: scene                 # 切片方式：scene=按镜头 / window=固定窗口
  scene_threshold: 27.0       # 镜头检测灵敏度（越小切得越碎）
  min_clip_seconds: 2.0       # 太短的片段丢弃
  max_clip_seconds: 30.0      # 太长的片段再切
  sample_fps: 2.0             # 过滤时每秒采样几帧做检测

filter:
  enabled_detectors: [death_screen, menu_shop, loading_black, motion]   # 启用哪些检测器
  bad_frame_ratio_to_drop: 0.25     # 坏帧占比超 25% 丢整片
  static_frame_ratio_to_drop: 0.6   # 静止帧占比超 60% 丢整片
  min_motion_magnitude: 0.6         # 光流幅度低于此视为"静止"
```

**改配置最常见的几种需求：**

- **想抓别的游戏/题材**：改 `queries`（关键词）和 `require_title_keywords`（标题白名单）。
- **想加自己精选的视频/合集**：往 `seed_urls` 里贴链接，直接重跑即可增量采集（无需 `--refresh-discovery`）。
- **磁盘小 / 电脑卡**：把 `download_concurrency`、`download_prefetch`、`segment_prefetch` 调成 1，`num_workers` 调小。
- **想要更多/更少素材**：改 `target_hours`（设 0 表示不限、处理完所有候选）。
- **过滤太严/太松**：调 `bad_frame_ratio_to_drop`、`static_frame_ratio_to_drop`、`min_motion_magnitude`。

## 8. 常见问题与排错

**Q：报错 `ModuleNotFoundError: No module named 'yt_dlp'`？**
A：你没激活对的环境。先 `conda activate gta_idm_data`，再跑。

**Q：YouTube 下载报 "Failed to get pixel format" 或解码失败？**
A：多半是 AV1 编码 OpenCV 解不了。工具已默认优先选 avc1（H.264）编码规避；若仍出现，
确认 ffmpeg 已正确安装。

**Q：发现阶段抓到的候选很少 / 报 429？**
A：被站点限流了。工具已内置随机等待 + 指数退避重试。可以等一会儿再跑，或降低
`download_concurrency`（YouTube 建议 2~3）、调大 `sleep_interval_s`。

**Q：磁盘很快就满了？**
A：把 `keep_raw` 设为 `false`（处理完即删原始片），并把 `download_prefetch` /
`segment_prefetch` 调成 1（减少同时在盘的视频数）。

**Q：跑着跑着好像卡住不动了？**
A：超长视频切片那一步本身就慢（会显示进度条）。默认配置已用 `max_duration_s: 10800`
拦掉 3 小时以上的视频来规避这个问题。若你把它设成 0（不限），就可能被超长视频拖死。

**Q：我中断了，重跑会不会重头来？**
A：不会。它会读 `processed.txt` 跳过已处理视频，读 `manifest.jsonl` 接着累计有效时长。
想完全重来就删掉 `data/` 目录（或换一个 `work_dir`）。

**Q：想重新搜索（不用缓存）怎么办？**
A：加 `--refresh-discovery`，或者直接删掉 `data/discovery_cache.jsonl`。
注意：只是想**新增** seed_urls 的话不需要这么做，直接重跑即可增量采集。

**Q：怎么提高过滤的精度？**
A：两个钩子——① 把死亡/菜单截图放进 `assets/templates/death/`、`assets/templates/menu/`
启用模板匹配；② 训练一个帧分类器，把权重路径填到 `filter.classifier_ckpt` 并在
`enabled_detectors` 里加上 `classifier`（接口见 [`filters/classifier.py`](gta_pipeline/filters/classifier.py)）。

## 9. 名词小词典

| 名词 | 大白话解释 |
|------|-----------|
| **pipeline（流水线）** | 把"发现→下载→切片→过滤→裁剪"几个步骤串起来自动跑完的整套流程 |
| **yt-dlp** | 一个开源的万能视频下载器，支持 YouTube/B 站等几百个网站 |
| **ffmpeg** | 处理音视频的命令行工具，负责合并、转码、裁剪、改帧率 |
| **OpenCV (cv2)** | 计算机视觉库，这里用来读帧、算图像特征（光流、梯度、亮度等） |
| **PySceneDetect** | 自动检测视频里"镜头切换"位置的库，用于按镜头切片 |
| **帧 / 帧率 (fps)** | 视频是一张张静止图片（帧）快速播放；每秒多少张就是帧率 |
| **抽帧采样** | 不逐帧看，每秒只取固定几帧来判断内容，省算力 |
| **光流 (optical flow)** | 估计相邻两帧之间像素"移动了多少"，用来衡量画面里有没有在动 |
| **去饱和 (desaturate)** | 颜色变淡、接近灰度。GTA 死亡画面会整屏去饱和 |
| **梯度 / 平坦区域** | 梯度衡量像素明暗变化的剧烈程度；菜单的纯色面板梯度小（平坦），游戏画面梯度大（有纹理） |
| **检测器 (detector)** | 专门判断"某帧是不是某类坏画面"的小模块，本工具有 4 个 |
| **manifest（清单）** | 记录每个保留片段元数据的文件，每行一个 JSON（JSONL 格式） |
| **断点续跑** | 中断后重跑能接着上次的进度，不重头来 |
| **进程 / 线程 / 并行** | 让多个任务同时跑、充分利用多核 CPU 和网络带宽 |
| **背压 (backpressure)** | 下游处理不过来时，用有界队列让上游自动慢下来，防止堆爆磁盘/内存 |
| **spawn / fork** | 两种启动子进程的方式；本工具必须用 spawn 避免多线程环境下的死锁 |
| **IDM / world model** | 让 AI 从视频里学习动作和环境物理规律的模型，是本数据集的服务对象 |
